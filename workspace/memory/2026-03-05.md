### 2026-03-05 10:10 AIFD 方案 Review 教训

**问题根因分析：**

1. **信息不足导致机制描述错误**：Agent Teams 的创建方式（自然语言 vs --agents 参数）、Hook 参数传递方式（stdin vs 命令行参数）、teammates 无法 resume 等关键限制，在写方案时未查阅官方文档验证，导致 MF-2 和 MF-4 这类"与实际不符"的问题。根因是**假设未验证**——基于对工具的模糊理解直接设计，没有 fact-check。

2. **设计粒度不够导致核心机制缺失**：跨 session 状态管理（MF-1）、大任务拆分（MF-3）、幂等恢复（SF-1）这些问题的共同特征是——方案只描述了"做什么"而没有描述"怎么做"。状态管理是分布式系统的核心问题，却完全没有设计。根因是**过早停留在架构层面**，没有往下钻一层思考执行细节。

3. **过度设计与不足并存**：质量把关设计了 5 阶段完整流程（过度），但跨 session 状态管理完全空白（不足）。说明精力分配不合理——在"锦上添花"的部分花了过多时间，忽略了"地基"问题。

**经验教训：**

1. **先验证再设计**：涉及外部工具/API 时，先写最小验证脚本确认实际行为，再基于事实设计。不要基于记忆中的模糊理解写方案。
2. **状态管理是第一优先级**：任何涉及多步骤、多 session 的系统，首先设计状态持久化和恢复机制。这是地基，不是后期优化。
3. **每个"原则"都要有对应的"机制"**：写"幂等设计"不够，必须写出 git checkpoint 这样的具体方案。如果写不出具体方案，说明还没想清楚。
4. **控制设计粒度**：先粗后细没问题，但核心路径（状态管理、错误恢复、任务编排）必须在第一版就有可执行的设计，不能留空。
5. **质量门要分级**：不是所有阶段都需要最重的流程。按风险和重要性分级，避免过度设计。

**改进措施：**

1. **技术方案 checklist**：写完方案后用 checklist 自查——外部依赖是否已验证？状态管理是否有具体设计？错误恢复路径是否覆盖？每个原则是否有对应机制？
2. **Review 前置验证**：关键技术假设（如 Hook 行为、API 限制）在写方案前就做 spike 验证，而不是写完再发现不对。
3. **Review 流程改进**：当前 Review 发现了问题但也暴露了初版方案质量不高的问题。未来可以在写方案时就做 self-review（换个角度审视），减少 Review 迭代。
